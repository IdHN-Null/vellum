# 설계 결정 기록 (Design Rationale)

`ARCHITECTURE.md`가 "무엇이, 어떻게" 동작하는지를 다룬다면, 이 문서는 **"왜 그렇게 만들었는가"**를 다룹니다. 각 결정마다 고려한 대안, 선택의 근거, 그리고 감수한 대가(trade-off)를 함께 적었습니다.

---

## 0. 요약 — 이 프로젝트를 관통하는 한 문장

> **"지형은 계산될 뿐 저장되지 않고, 좌표는 정규화되며, 화면은 순수 캔버스로만 그린다."**

거의 모든 결정이 이 세 가지 원칙에서 파생됩니다. 하나씩 그 이유를 풀어봅니다.

---

## 1. 기술 스택 — 무엇을 쓰고, 무엇을 안 썼나

### 쓴 것

| 기술 | 이유 |
|---|---|
| **TypeScript** | Obsidian API 자체가 TS로 타입이 제공됨. 지형·마커 같은 데이터 구조가 많아 컴파일 타임 검증의 이득이 큼 |
| **Canvas 2D API** | 지도는 결국 픽셀 덩어리. DOM 요소 수천 개보다 캔버스 한 장이 압도적으로 빠름 |
| **esbuild** | Obsidian 공식 플러그인 템플릿의 표준. 번들 속도가 밀리초 단위 |
| **Obsidian API만** | `TextFileView`, `Modal`, `Menu`, `FuzzySuggestModal` 등 앱이 제공하는 것을 최대한 재사용 |

### 의도적으로 **안 쓴** 것 — 여기가 더 중요합니다

**① three.js / WebGL을 쓰지 않았다**

3D를 넣는다면 three.js가 가장 쉬운 길입니다. 그런데도 순수 Canvas 2D 소프트웨어 렌더링을 택한 이유:

- **번들 크기.** three.js는 최소 수백 KB입니다. 우리 플러그인 전체가 `main.js` 64KB인데, 라이브러리 하나가 본체의 몇 배가 되는 건 "가벼운 3D"라는 요구와 정면으로 어긋납니다.
- **요구사항의 성격.** 사용자가 원한 건 "차별점을 위한 **가볍게 가능한** 3D"이지, 실시간 게임 엔진이 아닙니다. 하이트맵을 100×100 격자로 기울여 보여주는 정도는 회전 행렬 + 원근 나눗셈 + 페인터 알고리즘, 즉 고등학교 수준 수학 30줄이면 됩니다.
- **의존성 리스크.** 외부 라이브러리는 버전 충돌·보안 감사·Obsidian 업데이트 호환성 문제를 끌고 옵니다. 의존성 0개면 이 리스크가 통째로 사라집니다.

**대가:** 셰이딩·그림자·안티앨리어싱 같은 고급 표현은 포기했습니다. 격자를 촘촘히 하면 느려지므로 100×100으로 고정했습니다. 하지만 "지형의 기복을 입체로 확인한다"는 목적에는 충분합니다.

**② 캔버스 렌더링 라이브러리(PixiJS 등)도 안 썼다**

이유는 동일합니다. 우리가 그리는 건 배경 이미지 1장 + 마커 수십 개 + 폴리곤 몇 개뿐이라, 라이브러리의 씬 그래프·배칭이 필요 없습니다. `ctx.drawImage` + `ctx.fillText` 직접 호출이 더 단순하고 빠릅니다.

**③ 상태 관리 라이브러리(React 등)도 안 썼다**

Obsidian 뷰는 이미 명령형(imperative) DOM 모델입니다. 상태는 `this.map` 객체 하나이고, 바뀌면 `draw()`를 다시 부르면 됩니다. React를 얹으면 오히려 캔버스와 가상 DOM 사이에 불필요한 층이 생깁니다.

## 2. 왜 지형을 저장하지 않고 "레시피"만 저장하는가

이게 가장 중요한 아키텍처 결정입니다.

**대안 A — 생성된 이미지를 저장한다.** 지형을 만들어 PNG로 볼트에 저장.
**대안 B (채택) — 시드와 파라미터만 저장하고, 열 때마다 재생성한다.**

B를 택한 이유:

- **파일 크기.** 512×384 지형 PNG는 수백 KB~수 MB입니다. 반면 `{seed: 151186, seaLevel: 0.52, ...}`는 수백 바이트입니다. 볼트 100개 지도가 이미지면 무겁고, 레시피면 깃털처럼 가볍습니다.
- **버전 관리·동기화.** `.fmap`이 텍스트(JSON)라서 Git diff가 사람이 읽히고, Obsidian Sync가 효율적으로 처리합니다. 바이너리 이미지는 diff가 불가능하고 매번 통째로 재전송됩니다.
- **편집 가능성.** 슬라이더를 다시 움직이면 지형이 즉시 바뀝니다. 이미지였다면 "다시 생성" = 처음부터. 레시피 방식은 편집이 곧 파라미터 조정입니다.
- **결정론의 활용.** `mulberry32` 시드 난수가 결정론적이라, 같은 레시피는 어느 기기·어느 시점에서 열어도 **픽셀 단위로 동일한 지형**을 만듭니다. 재생성이 저장을 대체할 수 있는 근거입니다.

**대가:** 열 때마다 지형을 계산해야 합니다(50~90ms). 하지만 사람이 파일을 여는 빈도를 생각하면 무시할 수준이고, 그 대가로 위의 이득을 전부 얻습니다.

**절충 지점 — 브러시 편집만은 저장한다.** 손으로 칠한 지형 편집은 "레시피"로 환원할 수 없습니다(무작위가 아니라 사용자의 의도이므로). 그래서 이것만 `editsB64`로 저장하되, 원본 픽셀이 아니라 **기준 지형 대비 델타(고도 조정값)**만 `Int8Array`로 압축합니다. 편집이 없으면 필드 자체가 사라져 파일이 깨끗하게 유지됩니다.

## 3. 왜 정규화 좌표(0~1)인가

마커를 저장하는 방법은 두 가지입니다.

**대안 A — 픽셀 좌표.** `{x: 183, y: 92}`
**대안 B (채택) — 정규화 좌표.** `{x: 0.36, y: 0.42}` (지도 폭·높이의 비율)

B를 택한 이유는 이 플러그인이 **여러 좌표계를 넘나들기 때문**입니다.

- 지도 해상도가 512일 수도, 불러온 이미지가 4000픽셀일 수도 있습니다.
- 화면은 확대·축소·이동됩니다.
- 2D일 때와 3D 투영일 때 좌표가 완전히 다릅니다.

픽셀 좌표라면 이 모든 변환마다 마커 좌표를 다시 계산해 저장해야 하고, 이미지를 바꾸면 마커가 어긋납니다. 정규화 좌표는 **"지리적 위치"라는 불변의 진실**만 저장하고, 화면 픽셀은 그릴 때 그때그때 곱셈 한 번으로 구합니다.

이 결정의 파급효과는 큽니다 — **생성 지형과 이미지 모드가 마커·지역 코드를 100% 공유**할 수 있는 이유가 바로 이것입니다. 배경이 무엇이든 마커는 "폭의 36%" 지점이니까요.

## 4. 왜 `TextFileView`를 상속하는가

Obsidian에서 커스텀 화면을 만드는 방법은 여러 가지입니다.

- `ItemView` — 사이드바 패널 등 파일과 무관한 뷰
- `FileView` — 파일과 연결되지만 저장 로직은 직접
- **`TextFileView` (채택)** — 텍스트 파일의 읽기/저장 생명주기를 프레임워크가 관리

`TextFileView`를 고른 이유:

- `.fmap`은 본질적으로 텍스트(JSON) 파일입니다. `getViewData()`/`setViewData()`만 구현하면 **Obsidian이 저장 타이밍·더티 상태·충돌을 알아서 처리**합니다.
- `requestSave()`를 부르면 앱이 디바운스해서 저장합니다. 우리가 파일 I/O를 직접 관리할 필요가 없습니다.
- 파일 탐색기·탭·링크 등 Obsidian의 파일 기능이 공짜로 딸려옵니다.

즉 "지도를 파일처럼 다룬다"는 목표에 프레임워크가 가장 잘 맞춰주는 계약이 `TextFileView`였습니다.

## 5. 왜 "계산"과 "그리기"를 분리했나

`terrain.ts`(데이터)와 `render2d.ts`/`render3d.ts`(픽셀)를 엄격히 분리한 이유:

- **성능 — 재계산 회피.** 스타일만 바꾸면(양피지→잉크) 지형은 그대로고 색칠만 다릅니다. 분리돼 있으니 `terrain`을 건너뛰고 `render`만 다시 부릅니다. 만약 섞여 있었다면 색 하나 바꾸려고 노이즈를 다시 돌려야 했을 겁니다.
- **한 데이터, 두 표현.** 같은 `TerrainResult`를 2D 렌더러도 3D 렌더러도 소비합니다. 지형 로직이 화면을 모르기 때문에 가능한 재사용입니다.
- **테스트 용이성.** `terrain.ts`는 순수 함수(입력 → 출력)라 Obsidian 없이 Node에서 바로 검증됩니다. 실제로 강 생성 버그를 `test/debug-rivers.ts`로 앱 밖에서 잡았습니다.

이것은 고전적인 **관심사 분리 + 파이프라인 아키텍처**입니다: `noise → terrain → render → view`. 각 단계는 앞 단계의 출력만 알면 됩니다.

## 6. 지형 알고리즘 선택의 이유

**왜 Perlin 계열 그래디언트 노이즈 + fBm인가?**

- 순수 난수는 지형이 될 수 없습니다(인접 픽셀이 무관해서 TV 노이즈처럼 됨). 그래디언트 노이즈는 **인접 좌표가 부드럽게 이어져** 자연 지형처럼 보입니다.
- fBm(옥타브 쌓기)은 "큰 대륙 윤곽 + 중간 산맥 + 잔잔한 해안선"을 **하나의 파라미터 조절로** 표현하게 해줍니다. 슬라이더가 옥타브 수·감쇠율에 직접 매핑됩니다.
- Simplex 노이즈가 더 빠르지만, 특허·구현 복잡도 이슈가 있고 우리 해상도에선 Perlin으로 충분히 빠릅니다(50ms대).

**왜 `mulberry32`인가?** 시드→난수 생성기 중 가장 짧고(4줄) 품질이 충분합니다. 암호학적 강도는 필요 없고, "같은 시드 = 같은 지도"라는 결정론만 있으면 됩니다.

**강 생성 — 경사 하강 + 웅덩이 채우기.** 물리적으로 "물은 낮은 곳으로 흐른다"를 그대로 코딩했습니다. 문제는 노이즈 지형의 무수한 국소 웅덩이. 정석은 무거운 depression-filling 전처리(Planchon-Darboux 등)지만, 우리는 **물방울이 갇힐 때만 그 칸을 살짝 높이는** 경량 즉석 방식으로 타협했습니다. 완벽한 수문학은 아니지만 "지도에 강이 그럴듯하게 보인다"는 목표엔 충분하고 훨씬 빠릅니다.

## 6.5. 왜 3D를 격자 메시에서 벡터 등고선 테라스로 바꿨나

첫 구현은 하이트맵을 100×100 격자로 다운샘플해 1만 개 쿼드를 깊이 정렬하는 방식이었습니다. 동작은 했지만 두 가지 불만이 있었습니다: 2D(부드러운 픽셀 지형)와 3D(각진 격자)의 인상이 서로 달랐고, 매 프레임 1만 개 쿼드 정렬은 낭비였습니다.

**채택한 대안 — 등고선을 벡터로 추출해 z축으로 쌓기.**

- 마칭 스퀘어로 하이트맵에서 등고선 폴리곤을 추출하고(`contours.ts`), Douglas-Peucker로 단순화합니다.
- **같은 벡터 데이터가 2D와 3D 양쪽의 원천**이 됩니다. 2D에선 등고선 오버레이로, 3D에선 각 폴리곤을 해당 고도로 올려 벽으로 이은 계단식 테라스로. 두 뷰의 시각 언어가 일치합니다.
- 가림 처리가 극적으로 단순해집니다. 낮은 레벨부터 그리면 위에서 보는 카메라에선 자연히 맞고, 깊이 정렬은 레벨 내 벽 쿼드에만 필요합니다. 도형 수가 1만 → ~2천으로 줄어 렌더가 25ms → 7~8ms로 빨라졌습니다.
- "정밀도"라는 **하나의 파라미터가 벡터 세밀도를 직접 제어**합니다(레벨 수 6~14 + 단순화 강도). 격자 방식에선 이런 직관적 손잡이를 만들기 어려웠습니다.

**대가:** 지형이 계단식으로 양자화되어 부드러운 능선은 표현하지 못합니다. 하지만 이는 결점이 아니라 스타일입니다 — 레이저컷 지형 모형의 미학이고, 판타지 지도의 손맛과 잘 어울립니다.

기술 교훈: 등고선이 지도 경계에서 끊기면 3D에서 채울 수 없으므로, **지도 밖을 "매우 낮은 값"으로 취급해 모든 링이 강제로 닫히게** 했습니다. 섬 안의 호수 같은 구멍은 캔버스의 `evenodd` 채우기 규칙이 공짜로 처리합니다(이 규칙은 감김 방향을 무시하므로 링의 방향 일관성을 관리할 필요도 없어집니다).

## 7. 상호작용 설계의 이유

**왜 도구(tool) 상태 머신인가?** 마커 추가·지역 그리기·브러시·패닝은 모두 "마우스 드래그"라는 같은 입력을 씁니다. `this.tool` 하나로 분기하면, 이벤트 핸들러는 하나로 두고 도구별 동작만 갈아끼울 수 있습니다. 도구마다 별도 이벤트 시스템을 두는 것보다 단순합니다.

**왜 디바운스 재생성인가?** 슬라이더를 드래그하면 초당 수십 번 값이 바뀝니다. 매번 지형을 재계산하면 끊깁니다. 그래서 **"멈춘 뒤 잠깐 있다 한 번만"** 계산합니다. 브러시는 반응성이 중요해 짧게(60ms), 슬라이더는 여유 있게(200ms) 다르게 튜닝했습니다.

**왜 마커·라벨은 화면 좌표로 직접 그리나?** 지형은 카메라 변환(`translate`+`scale`) 안에서 그리지만, 마커 아이콘과 글자까지 스케일에 넣으면 확대할 때 글자가 거대해집니다. 그래서 지형만 변환 안에서, 텍스트·핀은 매번 화면 픽셀로 계산해 **두께·크기를 일정하게** 유지합니다.

**왜 오프스크린 캔버스에 지형을 캐시하나?** 매 프레임 픽셀 순회로 지형을 다시 칠하면 패닝만 해도 버벅입니다. 지형을 캔버스 한 장에 미리 그려두고, `draw()`에선 `drawImage`로 통째로 옮깁니다(GPU 가속). 지형이 실제로 바뀔 때만 이 캐시를 갱신합니다.

## 8. 데이터 연결(노트 링크) 설계의 이유

**왜 `notePath`를 마커 데이터에 직접 박나?** 마커와 노트를 잇는 방법은 여러 가지지만(별도 인덱스 파일, 노트 frontmatter 등), 마커 객체 안에 경로를 직접 넣는 것이 가장 단순하고 자기완결적입니다. `.fmap` 파일만 있으면 연결 정보가 온전합니다.

**왜 노트→지도는 "모든 .fmap을 훑는" 방식인가?** 역방향 인덱스(노트가 자기 마커를 아는 구조)를 유지하면 빠르지만, 두 곳의 정보를 동기화해야 하고 깨질 위험이 있습니다. 볼트의 `.fmap` 개수는 현실적으로 적으므로(수십 개), 명령 실행 시 `cachedRead`로 전수 검색하는 편이 **단일 진실 원천(single source of truth)**을 지키면서 충분히 빠릅니다. 캐시 무효화 버그를 원천 차단하는 선택입니다.

**왜 Obsidian의 `openLinkText`를 쓰나?** 직접 파일을 여는 대신 이 API를 쓰면 새 탭·분할·미리보기 등 사용자의 Obsidian 설정과 단축키가 그대로 존중됩니다. 플랫폼의 관례를 거스르지 않는 것입니다.

---

## 9. 두 모드(생성/이미지)를 한 코드로 합친 이유

이미지 모드는 별도 플러그인처럼 만들 수도 있었습니다. 하지만 **마커·지역·노트 링크·저장 로직이 완전히 동일**하기 때문에, `mode` 플래그 하나로 배경 소스만 분기하는 게 압도적으로 경제적입니다.

- 배경이 `generated`면 terrain 파이프라인, `image`면 Blob URL — 이 지점만 다릅니다.
- 나머지(정규화 좌표 덕분에) 전부 공유됩니다.

결과적으로 "직접 그린 지도에 마커 찍기"라는 폴백 기능이 **거의 공짜로** 딸려왔습니다. 정규화 좌표 결정(3장)이 여기서 배당금을 지급하는 셈입니다.

## 10. 감수한 대가들 (솔직한 한계)

좋은 설계 문서는 약점도 적습니다.

| 결정 | 얻은 것 | 잃은 것 |
|---|---|---|
| 순수 캔버스 3D | 의존성 0, 64KB | 고급 셰이딩·큰 지형에서의 부드러움 |
| 레시피만 저장 | 가벼운 파일, 버전 관리 | 열 때마다 재계산(50~90ms) |
| 전수 검색(노트→지도) | 동기화 버그 없음 | 지도 수가 수백 개면 느려질 수 있음 |
| CPU 픽셀 렌더링 | 단순함 | 초대형 해상도(4K+)는 부담 |
| 경량 강 알고리즘 | 빠름 | 엄밀한 수문학은 아님 |

이 대가들은 전부 **"세계관 창작자가 노트와 함께 쓰는 가벼운 지도 도구"**라는 목표에 비추어 의도적으로 받아들인 것입니다. 만약 목표가 "수백만 폴리곤 GIS 뷰어"였다면 정반대 결정(WebGL, 타일링, 공간 인덱스)을 했을 겁니다. 설계는 목표를 향한 선택이지, 절대적 정답이 아닙니다.

## 추기 — v0.4 ~ v0.11 결정 기록 (2026-07-07 ~ 07-11)

### 3D를 코어에서 제거하고 애드온으로 분리 (v0.10)
내장 소프트웨어 3D(하이트맵 메시)는 유지 비용 대비 만족도가 낮았습니다. **코어는 의존성 0을 유지**하고, 3D는 three.js를 쓰는 별도 애드온 플러그인으로 분리하기로 결정 (미착수). 코어의 벡터 등고선은 애드온의 입력이 됩니다.

### 잉크&워시 렌더 — "저해상도 색은 워시로, 디테일은 벡터로" (v0.7~)
셀 해상도 비트맵을 확대하면 뿌옇게 됩니다(Wonderdraft도 동일). 해법은 ESRI 고지도 기법: **바탕색은 수채 워시로 남기고, 선명해야 하는 것(해안선·등고선·강·글리프)은 벡터/초과표본으로** 위에 얹기. 여기에 화면 해상도 종이결 오버레이로 배율 무관 질감을 더함.

### 3x 상세 캐시 + 프로그레시브 타일 (v0.8~0.10)
확대 선명도(LOD)와 무정지 렌더를 모두 잡기 위해: 지형 확정 시 전체 맵을 셀당 3px로 백그라운드 재렌더 → pan/zoom은 캐시 크롭만. 최초 표시는 192셀 타일을 뷰포트 가까운 순으로 프레임당 10ms씩. **대가**: base 픽셀 규칙과 LOD 픽셀 규칙을 두 곳에서 동일하게 유지해야 함 (렌더 수정 시 주의).

### 대륙 등뼈 산맥 보장 (v0.11)
플레이트오 falloff + 절대 스케일 클램프로 "평야 우세"를 지키되, 노이즈 게이트만으로는 **산맥이 전혀 없는 대륙**이 나올 수 있었습니다(시드 편차). 전역 게이트를 여는 대신 **대륙 블롭마다 능선 세그먼트(spine) 1줄기를 명시적으로 보장** — 판타지 지도 문법(대륙엔 산맥 등뼈)을 시드 운에 맡기지 않습니다. 노이즈 산맥 파라미터는 원래 균형을 유지.

### 해안 동심 잔선을 벡터가 아닌 픽셀 필드로 (v0.11)
고지도의 등거리 잔물결 링은 waterDistance BFS 필드에서 거리 기반 가우시안 밴드로 그립니다. 벡터 등고선 추출보다 훨씬 싸고, 브러시 dirty-rect 갱신과도 자연스럽게 맞물립니다. LOD에서는 같은 필드를 쌍선형 샘플.

### PNG 내보내기 = 상세 캐시 재사용 (v0.11)
내보내기가 셀 해상도 base를 업스케일해 편집기 화면보다 뿌옇던 문제를, 유효한 3x 상세 캐시를 그대로 그려 해결. **WYSIWYG 원칙**: 편집기에서 보는 것이 곧 내보내는 것.

### 데스크톱 전용 선언 (v0.11)
번들 폰트로 styles.css가 400KB이고 모바일 검증이 없어 `isDesktopOnly: true`로 정직하게 선언. 모바일 검증 후 되돌릴 수 있음.
