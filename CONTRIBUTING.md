# 기여 가이드 — 브랜치 방침

이 저장소는 아래 5종 브랜치로 운영합니다. 요약 한 줄:

> **코드는 `feat/*`에서만 쓰고, `dev`로 모으고, `master`는 검증된 결과만 받는다.**

## 1. `master` — 배포용

- **통합 + 검증(빌드·테스트)이 모두 끝난 버전만** 존재해야 합니다.
- 여기서 직접 개발하지 않습니다. 커밋은 `dev`(또는 `hotfix/*`)에서의 병합으로만 생깁니다.
- `master`가 곧 배포본입니다 — `dist/`의 릴리스 zip과 GitHub Pages(`docs/`)가 이 브랜치를 기준으로 서빙됩니다.

## 2. `dev` — 통합(integration)용

- 이름은 개발이지만 실제 역할은 **통합**입니다. **여기서 코드를 직접 작성하지 않습니다.**
- 별도로 개발한 기능을 합치고 싶으면 `feat/*` 브랜치에서 **`dev`로 PR**을 넣습니다.
- `dev`에서 충분히 안정화되면 `master`로 병합(PR)합니다.

## 3. `feat/{기능명}` — 개별 기능 개발용

- `dev`에서 분기해서 하고 싶은 것을 자유롭게 개발합니다.
- **PR을 넣기 전에 반드시**, 로컬에서 `dev`를 병합해 충돌을 해결하고
  `npm run build && npm test`가 통과하는지 확인합니다.
- **자기 PR 자체 승인 금지.** (리뷰를 거치지 않은 병합 금지)
- PR이 승인·병합되면 해당 브랜치는 **자동 삭제**됩니다 (저장소 설정).
- 쩔거나 재밌는 기능, 연습 결과를 자랑만 하고 싶고 합치기는 애매하다면 —
  push만 해두고 PR을 넣지 않아도 됩니다.

## 4. `test/{테스트명}` — 실험용

- 실험·검증 전용. **PR 금지.**
- 결과가 쓸 만하면 `feat/*`로 옮겨 정식으로 개발합니다.

## 5. `hotfix/{오류명}` — 긴급 수정용

- `dev`나 `master`에서 **긴급한 문제**가 발생했을 때 해당 브랜치에서 분기합니다.
- 완전히 해결한 뒤 원래 브랜치로 병합합니다 (`master`에 적용했다면 `dev`에도 반영).
- 다른 브랜치(`feat/*` 등)에서 생긴 문제는 hotfix 대상이 아닙니다 — 그 브랜치에서 직접 해결하세요.

---

## PR 체크리스트

- [ ] 로컬에서 `dev` 병합 → 충돌 해결 완료
- [ ] `cd plugin && npm run build` 통과 (타입체크 + 번들)
- [ ] `npm test` 통과
- [ ] 렌더링에 영향 주는 변경이면 시각 하네스(`node test/server.mjs`)로 확인
- [ ] 자체 승인 아님

## 흐름 요약

```
feat/멋진기능 ──PR──▶ dev ──PR──▶ master ──▶ dist/ 릴리스 + docs/ 페이지
     ▲                 │
     └── 분기는 여기서 ─┘          hotfix/급한버그 ──▶ (발생한 브랜치로 병합)
```
