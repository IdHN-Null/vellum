# コントリビュートガイド — ブランチ方針

🌐 [English (US)](CONTRIBUTING.md) · [English (UK)](CONTRIBUTING.en-GB.md) · [Español](CONTRIBUTING.es.md) · [中文](CONTRIBUTING.zh.md) · **日本語** · [한국어](CONTRIBUTING.ko.md)

このリポジトリは以下の 5 種類のブランチで運用します。一行で言うと：

> **コードは `feat/*` でだけ書き、`dev` に集め、`master` は検証済みの結果だけを受け取る。**

## 1. `master` — リリース用

- **統合 + 検証（ビルド・テスト）が完了したバージョンだけ**が存在できます。
- ここでは直接開発しません。コミットは `dev`（または `hotfix/*`）からのマージのみ。
- `master` がそのままリリースです — `dist/` のリリース zip と GitHub Pages（`docs/`）はこのブランチ基準で配信されます。

## 2. `dev` — 統合用

- 名前は開発ですが、実際の役割は**統合**です。**ここで直接コードを書かないでください。**
- 別で開発した機能を取り込みたいときは、`feat/*` ブランチから **`dev` へ PR** を出します。
- `dev` が十分に安定したら、PR で `master` にマージします。

## 3. `feat/{機能名}` — 機能開発用

- `dev` から分岐して、自由に開発します。
- **PR を出す前に必ず**、ローカルで `dev` をマージして競合を解決し、
  `npm run build && npm test` が通ることを確認します。
- **自分の PR の自己承認は禁止。**（レビューなしのマージ禁止）
- PR が承認・マージされたブランチは**自動削除**されます（リポジトリ設定）。
- すごい機能や面白い実験を、マージはせず見せびらかしたいだけなら —
  push だけして PR を出さなくても構いません。

## 4. `test/{テスト名}` — 実験用

- 実験・検証専用。**PR 禁止。**
- 結果が使えそうなら `feat/*` に移して正式に開発します。

## 5. `hotfix/{バグ名}` — 緊急修正用

- `dev` や `master` で**緊急の問題**が起きたとき、そのブランチから分岐します。
- 完全に解決してからマージします（`master` に適用したら `dev` にも反映）。
- 他のブランチ（`feat/*` など）の問題は hotfix の対象外 — そのブランチで直接解決してください。

---

## PR チェックリスト

- [ ] ローカルで `dev` をマージ → 競合解決済み
- [ ] `cd plugin && npm run build` が通る（型チェック + バンドル）
- [ ] `npm test` が通る
- [ ] レンダリングに影響する変更はビジュアルハーネス（`node test/server.mjs`）で確認済み
- [ ] 自己承認ではない

## フロー概観

```
feat/すごい機能 ──PR──▶ dev ──PR──▶ master ──▶ dist/ リリース + docs/ ページ
     ▲                   │
     └── 分岐はここから ──┘          hotfix/緊急バグ ──▶ （発生したブランチへマージ）
```
